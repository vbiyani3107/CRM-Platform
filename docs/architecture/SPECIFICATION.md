## 1. Bespoke Schema Specifications (PostgreSQL)

The following tables constitute the foundational database specification. They map directly to the four luxury domains and rely heavily on `JSONB` and `vector` extensions.

### 1.1 Domain: Clienteling Hub

#### Table: `hyper_profiles`
Replaces generic "Accounts" and "Contacts". Manages the complex, evolving identity of an HNWI.

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE hyper_profiles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    full_name VARCHAR(255) NOT NULL,
    vip_tier VARCHAR(50) DEFAULT 'Standard',
    lifestyle_persona TEXT,     -- AI-maintained summary of tastes and habits
    influence_score NUMERIC(5,2), -- Calculated metric of the client's tastemaker status
    communication_protocols JSONB, -- Strict rules (e.g., "WhatsApp only, no calls")
    significant_dates JSONB,    -- Key milestones, anniversaries, family birthdays
    tastemaker_network JSONB,   -- Graph edges to other profiles they influence
    taste_vector vector(1536),  -- Continuously shifting math representation of aesthetic preference
    custom_attributes JSONB DEFAULT '{}', -- Dynamically extended by the Foundation service
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Index for high-speed AI similarity matching against The Vault (1536 dims for OpenAI embeddings)
CREATE INDEX ON hyper_profiles USING hnsw (taste_vector vector_l2_ops);
-- Index for extremely fast querying of dynamic JSONB schema attributes
CREATE INDEX ON hyper_profiles USING GIN (custom_attributes);
```

#### Table: `client_engagements`
Replaces generic "Activities". The source of truth for the Ambient Intelligence engine.

```sql
CREATE TABLE client_engagements (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    profile_id UUID REFERENCES hyper_profiles(id) ON DELETE CASCADE,
    engagement_type VARCHAR(50), -- e.g., 'Private Dinner', 'WhatsApp', 'Boutique Visit'
    raw_notes TEXT NOT NULL,     -- Associate's unstructured input
    ai_extracted_insights TEXT,  -- Structured extraction generated by LangChain4j
    engagement_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    custom_attributes JSONB DEFAULT '{}'
);
```

### 1.2 Domain: The Vault & Atelier

#### Table: `curated_vault`
Replaces generic "Products" or "Inventory". Stores high-end, often unique pieces.

```sql
CREATE TABLE curated_vault (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    collection_name VARCHAR(150),
    piece_name VARCHAR(255) NOT NULL,
    sku_or_serial VARCHAR(100) UNIQUE,
    material_composition JSONB, -- Deep dive into materials (e.g., 18k Rose Gold, VVS1 Diamonds)
    aesthetic_description TEXT, 
    aesthetic_embedding vector(1536), -- Vectorized description for semantic "Vibe" search
    current_location VARCHAR(100), -- e.g., 'Paris Flagship', 'Geneva Vault', 'With Client'
    custom_attributes JSONB DEFAULT '{}',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX ON curated_vault USING hnsw (aesthetic_embedding vector_l2_ops);
CREATE INDEX ON curated_vault USING GIN (custom_attributes);
```

#### Table: `provenance_ledger`
Replaces generic asset tracking. Stores immutable records of authenticity and digital twins.

CREATE TABLE provenance_ledger (

## 2. API Conventions

The API layer is exposed via Spring Boot and relies on the Foundation metadata engine to route requests to the correct dynamic tables.

### 2.1 Standard Dynamic Operations

| Method | Endpoint | Purpose |
|---|---|---|
| `GET` | `/api/v1/entities/{entityName}` | List/Filter records (SQL filtering over standard and JSONB fields) |
| `GET` | `/api/v1/entities/{entityName}/{id}` | Retrieve a single record |
| `POST` | `/api/v1/entities/{entityName}` | Create a record |
| `PATCH` | `/api/v1/entities/{entityName}/{id}` | Partially update a record |

### 2.2 Multi-Modal Semantic Aesthetic Search

Used exclusively for finding matches based on aesthetic, mood, or "vibe", supporting both text and image inputs.

#### Text-Based Search
```http
POST /api/v1/semantic-search/curated_vault/text
Content-Type: application/json

{
  "aesthetic_query": "Minimalist, quiet luxury, platinum undertones",
  "limit": 3
}
```

#### Image-Based Search (Visual Match)
```http
POST /api/v1/semantic-search/curated_vault/image
Content-Type: multipart/form-data

image=@client_inspiration.jpg
limit=3
```

**Execution Flow (LangChain4j + Vision Models + PostgreSQL):**
1. LangChain4j translates the text query OR the Vision Model (e.g., CLIP) translates the image into a unified `[1536-dimensional array]`.
2. The Database executes:
   ```sql
   SELECT id, piece_name, aesthetic_description 
   FROM curated_vault 
   ORDER BY aesthetic_embedding <-> '[embedded_vector]' LIMIT 3;
   ```

## 3. Metadata Engine Specifications

To support the UI and API dynamically adapting to luxury requirements, `sys_dictionary` must support exact types.

**`sys_dictionary`**
```sql
CREATE TABLE sys_dictionary (
    id UUID PRIMARY KEY,
    entity_name VARCHAR(50), -- e.g., 'hyper_profiles'
    attribute_name VARCHAR(50), 
    data_type VARCHAR(20), -- ENUM: 'string', 'boolean', 'jsonb', 'vector', 'reference'
    is_custom BOOLEAN,
    is_sensitive BOOLEAN -- If true, requires specific JWT roles to read/write (Discretion framework)
);
```

### 3.4 Data Hydration & Seed Strategy

A luxury platform cannot be developed effectively without a rich, semantic data set. We will use a script-driven approach to populate the database for local development:
1. **Catalog Seed**: Create at least 50 `curated_vault` items covering distinct aesthetic dimensions (e.g., minimalist watches, avant-garde jewelry) using generated embeddings.
2. **Profile Seed**: Generate 20 distinct `hyper_profiles` (HNWIs) with varying `taste_vector` values and complex `lifestyle_persona` records.
3. **Execution**: This will be driven by a custom Spring Boot `CommandLineRunner` or Python seed script using the LangChain4j model for realistic embeddings.

## 4. White-Glove Automation Contracts

### 4.1 Proactive Gifting (Spring Batch + Vectors)

A scheduled job designed to empower associates to surprise and delight HNWIs.
- **Schedule**: Daily at 02:00 AM Boutique Local Time.
- **Logic**: 
  1. Find `hyper_profiles` where `significant_dates` contains an event exactly 21 days from today.
  2. Read the client's `taste_vector`.
  3. Query `curated_vault` ordered by similarity to the `taste_vector`.
  4. Generate an alert in the Clienteling Hub for the assigned associate, suggesting the top 3 pieces.
