## What Was Built

The Concierge & Experiential Management hub provides luxury experiential request tracking, proactive gifting alerts powered by AI-driven taste matching, and ambient intelligence event processing.

### Core Entities

| Entity | Table | Description |
|--------|-------|-------------|
| `ExperientialRequest` | `experiential_requests` | Luxury client requests (Yacht Charter, Private Viewing, etc.) with discretion levels and JSONB custom attributes |
| `ConciergeAlert` | `concierge_alerts` | Proactive alerts generated by the daily batch job, containing AI-suggested vault items |

### API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/v1/concierge/requests` | List experiential requests (paginated) |
| GET | `/api/v1/concierge/requests/{id}` | Get request details |
| POST | `/api/v1/concierge/requests` | Create experiential request |
| PATCH | `/api/v1/concierge/requests/{id}` | Update request (partial) |
| POST | `/api/v1/concierge/requests/{id}/fulfill` | Mark as fulfilled |
| GET | `/api/v1/concierge/alerts` | List unread alerts for associate |
| POST | `/api/v1/concierge/alerts/{id}/read` | Mark alert as read |

## Decisions Made

1. **ConciergeAlert does NOT extend BaseEntity** — The spec schema only has `id` and `created_at`; no `updated_at`, `version`, or `is_deleted`. Extending BaseEntity would force unnecessary columns.

2. **`@Scheduled` instead of Spring Batch** — The prompt mentions "Spring Batch job" but the actual scan logic is a simple daily cron. Using `@Scheduled` avoids the Spring Batch dependency overhead while matching the intent. The method signature is designed for easy migration to Batch if needed.

3. **Stub mode for AI services** — The proactive gifting job's semantic search (taste_vector ↔ aesthetic_embedding) requires HyperProfile and CuratedVault entities from WP-HUB-01/02. The job is stubbed with clear TODO comments showing the exact SQL queries for production.

4. **Event-driven ambient intelligence** — All write operations fire `ConciergeRequestEvent` via Spring's `ApplicationEventPublisher`, consumed asynchronously by `ConciergeEventListener`. This decouples the API from LangChain4j processing.

5. **EXP number auto-generation** — Uses a `getNextSequenceNumber()` repository method that reads `MAX(CAST(SUBSTRING(number, 4)))` to avoid database sequences and maintain portability.
